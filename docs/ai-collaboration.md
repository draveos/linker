# AI와의 개발 협업 기록

이 문서는 Linker가 Claude Code (Opus 4.6)와의 지속적 대화를 통해 어떻게 구현되었는지 기록합니다. 공모전의 "AI와 효율적으로 협업하는 능력" 심사 기준에 대한 증거 자료입니다.

## 1. 협업 모델

인간의 역할:
- 비전 제시 ("오답의 근본 원인을 추적하는 메타인지 도구")
- 제품 결정 (UX 플로우, 우선순위, 트레이드오프 최종 판단)
- 도메인 지식 (교육 문제, 사용자 맥락)
- 피드백 & QA ("이 부분 아쉬워", "여기서 버그 있어")

Claude Code의 역할:
- 구현 설계 제안 (3-4가지 옵션 + 트레이드오프)
- 코드 작성 및 수정 (다중 파일 동시)
- 타입 체크 피드백 루프
- 디버깅 & 리팩터링
- 문서화

## 2. 세션 구조 (패턴)

대부분의 세션이 다음 3단계 루프로 진행됨:

### Step 1: 문제 진술
사용자가 자연어로 원하는 것 설명:
> "노드 드래그한 걸 저장되게 하고 싶은데, 리프레시하면 날아감"

### Step 2: 플래닝 대화
Claude가 3-4가지 접근 방식을 제시 + 각 트레이드오프:
> "Option A (정규 저장): 캔버스 저장 버튼 클릭 시만 persist. 장점: 예측 가능. 단점: 사용자가 저장을 깜빡함.
> Option B (자동 저장): 드래그 종료 시마다 persist. 장점: 놓치지 않음. 단점: 매 드래그마다 write.
> Option C (debounce): 드래그 후 500ms 디바운스. 장점: 균형. 단점: 복잡."

사용자가 선택: "C로 가자"

### Step 3: 구현 + 검증
Claude가 여러 파일 동시 수정:
- `lib/graph-store.ts` — 타입 확장
- `app/learn/page.tsx` — 핸들러 추가
- `components/knowledge-graph-canvas.tsx` — 콜백 wiring

즉시 `npx tsc --noEmit` 실행 → 에러 있으면 자가 수정.

## 3. 주요 설계 결정 (협업으로 내려진)

### 결정 1: Multi-Agent Harness 도입 이유와 범위

배경: 사용자가 "인턴 시 대표가 harness 시도해보라고 함" 언급.

Claude의 질문: "Harness를 프로젝트의 핵심 축으로 할까요, 아니면 기술 보너스로 할까요?"

논의:
- 핵심 축으로 하면 내러티브가 "멀티 에이전트 AI" 쪽으로 쏠림 → 교육 문제 해결이 희석됨
- 기술 보너스로 하면 "AI 활용 능력" 점수 + 오버인덱싱 방지

결정: 기술 보너스. 교육 문제 해결이 메인 내러티브, Harness는 차별화 포인트.

### 결정 2: 퀴즈 게이트 로직

초기 요구: 사용자가 "결손 노드 학습 완료 버튼이 바로 완료로 넘어가는데, 퀴즈 통과해야 하게 하자"

Claude의 명확화 질문: "퀴즈 통과 후 상태는 '완료'인가요, '학습 필요'인가요?"

사용자 답: "학습 필요. 완료는 따로 눌러야 함."

상태 머신 합의:
```
결손 → (퀴즈 통과) → 학습 필요 → (수동 클릭) → 완료
                                ← (토글) ←
```

### 결정 3: 토큰 최적화 - Two-phase Proposer

문제: 루프마다 Proposer가 마이크로 러닝 콘텐츠까지 생성 → 3라운드 시 콘텐츠 3번 중복 생성 → 2400tok 낭비

Claude의 제안:
- "루프 중엔 진단만 (100tok)"
- "루프 종료 후 Content Generator를 분리해서 1회만 호출 (800tok)"
- 이유: 재추론 시 노드가 바뀌면 이전 콘텐츠는 어차피 버려짐

결과: 평균 토큰 ~37% 절감.

### 결정 4: SSE vs WebSocket

Claude의 권장: SSE
- 단방향 스트리밍으로 충분
- HTTP 기반 → Vercel serverless 호환
- 재연결 로직 간단

트레이드오프: WebSocket은 양방향 가능하지만, 우리 케이스엔 불필요.

### 결정 5: 노드 색상 팔레트 ("얕은 네온")

사용자 피드백: "글래스모피즘 대신 얕은 네온으로"

Claude의 해석 → 합의:
- 배경: `bg-slate-900` (불투명 dark)
- 보더: `border-2 border-{color}-500/50` (얕은 네온)
- 글로우: `shadow-[0_0_Npx_rgba(...)]` (subtle)
- 텍스트: `text-{color}-300`

이 팔레트가 프로젝트 전반에 일관 적용됨 → CLAUDE.md에 기록.

## 4. 디버깅 세션 예시

사용자: "엣지 선이 안 보이고, 빨간 엣지는 노드랑 겹쳐서 화살표가 안 보여"

Claude의 진단 프로세스:
1. `knowledge-graph-canvas.tsx`의 SVG 엣지 렌더링 읽음
2. 문제 1: `stroke: "hsl(var(--border))"` — CSS 변수가 SVG inline style에서 resolve 안 됨 → 직접 hex 사용으로 변경
3. 문제 2: 엣지가 노드 중심에서 중심으로 그려짐 → 노드 크기만큼 겹침
4. 해결: unit vector 계산 후 양 끝 단축 (`startPad = 5%`, `endPad = 7%`)

사용자 피드백: "빨간 엣지도 마찬가지로 적용해줘"

Claude의 수정: 빨간 엣지의 arrow marker가 더 크므로 `endPad = 8%`로 추가 여유.

## 5. 스코프 제어

사용자가 여러 번 한 말: "지금 할 것만, 나중 거는 미뤄"

Claude의 대응 원칙:
- 요청한 것만 구현, 주변 리팩터링 금지
- "추후 개선" 아이디어는 제안하되 구현 안 함
- 오버 엔지니어링 자제

실제 적용 예:
- 그래프 생성에 Critic 루프 추가 제안 → 사용자 "시간 없으니 보류" → 구현 안 함
- Teacher dashboard 제안 → 사용자 "v2 로드맵으로" → 문서에만 언급

## 6. 긴 세션의 컨텍스트 관리

### 문제
개발 세션이 여러 시간 지속되면서 Claude 컨텍스트가 가득 참.

### 해결 전략
1. 중간 요약: 주요 마일스톤마다 "지금까지 한 거" 요약
2. CLAUDE.md 업데이트: 새로운 컨벤션 발견 시 즉시 기록
3. 명확한 참조: "knowledge-graph-canvas.tsx의 saveChanges 함수" 처럼 구체적 위치 언급

## 7. AI 협업의 정량 지표 (추정)

| 항목 | 값 |
|---|---|
| 주요 파일 수 | ~20개 |
| Claude가 생성한 LoC (추정) | ~3500+ |
| 수동으로 작성한 LoC | ~200 (주로 컨텐츠, 상수) |
| 세션 수 | ~30+ |
| 주요 리팩터링 | 5회 (Harness 도입, Two-phase proposer, 노드 상태 로직, 그래프 저장, landing mockup) |

## 8. 교훈

### 잘 작동한 것
1. "이게 왜 필요한지" 먼저 논의 — 코드 바로 쓰지 않음
2. 타입 체크 피드백 루프 — 에러 즉시 발견
3. CLAUDE.md로 컨벤션 고정 — 매 세션 재설명 불필요
4. 트레이드오프 명시 — 단순 "이게 좋음"이 아닌 "A는 빠르지만 B는 안전"

### 어려웠던 것
1. 긴 세션에서 과거 결정 잊기 — 컨텍스트 요약으로 완화
2. 오버 엔지니어링 경향 — 명시적 "현재 요구만" 지시 필요
3. 시각적 피드백 한계 — 코드는 잘 써도 "눈으로 보기에 어떤지"는 사용자 QA 필요

## 9. 결론

Linker는 Claude Code가 단순 코드 제너레이터가 아닌 설계 파트너로 작동했기에 가능했습니다. 인간은 비전과 판단을, AI는 구현과 검증을 담당하는 역할 분담이 핵심이었습니다.

특히 Multi-Agent Harness라는 메타 주제는 이 프로젝트의 자기 참조적 구조를 잘 보여줍니다 — AI 에이전트들이 협업해 학생을 돕는 도구를, 인간과 AI 에이전트가 협업해 만든 것입니다.
