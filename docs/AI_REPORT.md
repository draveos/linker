프로젝트명: Linker — AI 지식 그래프 기반 학습 결손 추적 솔루션

## 1. 기획

### ■ 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

**주요 사용자**

- **1차**: 자기주도 학습하는 중·고등학생, 대학생, 자격증 수험생 (특히 수능 수학, 공학수학, 알고리즘 같은 개념 의존성이 강한 과목)
- **2차**: 1:1 진단 시간이 부족한 학원/과외 강사, 교육 운영자

**구체적인 페인 포인트**

**① 같은 실수를 반복하는데, 학생도 교사도 근본 원인을 모름**

한 학생이 행렬 A의 역행렬을 계산하다가 부호를 틀립니다. "계산 실수네"로 넘어가지만 2주 뒤 고유값 문제에서 또 행렬식이 나오면 또 틀립니다. 학생 본인도, 교사도 이 학생이 **"행렬식의 부호 규칙에 결손이 있다"**는 진짜 원인을 인지하지 못합니다. 표면 증상(역행렬 오답)만 보고 근본 원인(행렬식 이해 부족)은 놓칩니다.

**② 학습 개념들 사이의 의존 관계가 머릿속에 그려지지 않음**

학생은 "역행렬이 안 되니 역행렬을 복습하자"고 판단하지만, 실제로는 역행렬의 선행 개념인 **행렬식**에 결손이 있는 경우가 대부분입니다. 개념 의존성 그래프가 머릿속에 없으면 **학습 우선순위를 잘못 세우게 되고**, 시간을 낭비합니다.

**③ 기존 AI 튜터(ChatGPT/Gemini)의 단방향·단발적 한계**

"역행렬이 안 풀려요"라고 물으면 그 문제의 답만 줍니다. 다음 질문은 처음부터 시작이고, 학생의 누적 약점 패턴을 기억하지 못합니다. **"왜" 이 학생이 반복해서 이 문제에서 막히는지, 그래프의 어느 노드가 근본 원인인지 추적해주지 않습니다.**

**④ 교사의 1:1 진단 부담**

30명 학급에서 각 학생의 오답 근본 원인을 짚어주는 데 한 명당 5-10분이 걸립니다. 자습 시간에는 불가능하고, 수업 시간에도 비효율적입니다. 교사는 "누가 어디서 막히는지"를 체계적으로 파악할 도구가 없습니다.

**Linker가 집중하는 핵심 문제**

> **"왜 틀렸는지 근본 원인을 못 찾아서 같은 실수를 반복하는 것"**

### ■ 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

Linker는 다음 6개 핵심 기능으로 구성됩니다.

**① 지식 그래프 시각화**

학습 도메인(선형대수학, 미적분학 등)의 개념들과 그들 사이의 선행 관계를 DAG(방향 있는 비순환 그래프)로 표현합니다. 학생은 "벡터 → 행렬 → 행렬식 → 역행렬"이라는 의존 체인을 시각적으로 파악합니다. 완료 개념은 파란색, 학습 필요는 회색, 결손은 빨간색으로 구분됩니다.

**② AI 오답 역추적**

학생이 오답 텍스트(잘못 푼 풀이, 이해 안 되는 부분 등)를 입력하면 AI가 지식 그래프를 역방향으로 순회하며 **가장 근본적인 결손 노드**를 특정합니다. 표면 증상(역행렬 계산 오류)이 아닌 **근본 원인(행렬식 부호 규칙 미숙)**을 찾는 것이 차별점입니다. 분석 완료 후 traversal path가 애니메이션으로 시각화됩니다.

**③ Multi-Agent Harness (멀티 에이전트 협업 루프)**

단일 AI 호출은 틀릴 수 있습니다. Linker는 **3개 전문 에이전트의 협업 루프**를 사용합니다:

- **Proposer (Claude Haiku 4.5)**: 그래프 역추적으로 근본 원인 후보 제안
- **Verifier (Claude Haiku 4.5)**: 제안을 비판적으로 검토, 더 근본적 원인이 있는지 검증
- **Content Generator (Claude Haiku 4.5)**: 확정된 결손 개념에 대한 학습 콘텐츠 생성

확신도(confidence)가 0.8 미만이면 Verifier가 투입되고, 반박 시 Proposer가 critique를 반영해 재추론합니다. 최대 3라운드 루프. 조기 종료 4가지(확신도 충분/Verifier 동의/수렴 감지/max rounds)로 토큰 낭비 방지.

**④ 3분 마이크로 러닝 + 퀴즈 게이트**

결손 개념에 대해 AI가 자동으로 3분 분량의 핵심 설명 + 4지선다 확인 퀴즈를 생성합니다. 핵심 차별점은 **"결손 개념은 퀴즈 정답 후에만 학습 완료 버튼이 활성화"**된다는 것입니다. 이해 확인 없이 "대충 알겠다"로 넘어가는 메타인지 함정을 막습니다.

**⑤ 학습 프로파일 누적**

모든 분석 결과(오답 텍스트, 근본 원인 노드, confidence, 에이전트 trace)가 개인 로그에 저장됩니다. 학생은 "내가 행렬식에서 3번, 역행렬에서 2번 막혔구나"라는 **개인 약점 패턴**을 시간이 지날수록 풍부하게 확보합니다. 단순 Q&A가 아닌 **"내 학습의 기억"**을 만드는 것이 차별점.

**⑥ 강의 텍스트 → 그래프 자동 생성**

자기 교재나 강의 노트가 있으면 텍스트를 입력해 Claude Sonnet 4.6이 자동으로 지식 그래프를 생성합니다. 입력이 100자 미만이면 Haiku가 추가 질문(최대 3회)으로 컨텍스트를 보강한 뒤 Sonnet 1회 호출로 그래프 품질을 확보합니다.

### ■ 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

**학생 관점**

1. **동일 패턴 재발 방지**
   근본 원인을 찾아 해결하므로 "같은 실수 반복" 감소. 표면 증상을 고치는 것과 근본 개념을 이해하는 것의 차이.

2. **학습 메타인지 강화**
   자기 약점을 그래프 위에서 시각적으로 인지. "나는 어디서 막히는 사람인지"가 명확해짐.

3. **학습 시간 효율화**
   3분 마이크로 러닝으로 핵심만 복습. 1시간 영상 시청의 대체재.

4. **자기주도 학습력 향상**
   교사 개입 없이 **진단 → 처방 → 이해 검증** 사이클이 앱 안에서 자동으로 돌아감.

**교사 관점**

1. **1:1 진단 부담 경감**
   오답 근본 원인 추적을 AI가 대신. 교사는 학생과 대화에 집중.

2. **학급 단위 약점 파악** (v2 로드맵)
   여러 학생의 로그 집계 시 "이 학급은 행렬식에 약하다" 같은 패턴 발견 가능.

3. **개별 맞춤 과제 자동 배포** (v2 로드맵)
   학생의 결손 노드 기반으로 관련 문제를 자동 배포.

**측정 가능한 정량 지표**

- 분석 횟수 / 숙련도 % / 라운드당 평균 토큰 / 검증 루프 발동 비율
- 이 지표들은 앱 내에서 실시간 표시되며, **교육 효과 연구의 데이터 소스로도 활용 가능**

**기존 솔루션 대비 차별점**

| 기능 | ChatGPT/Gemini | 기존 LMS | **Linker** |
|---|---|---|---|
| 학습 구조 시각화 | ❌ | 🟡 정적 커리큘럼 트리 | ✅ 동적 DAG |
| 근본 원인 추적 | ❌ 질문별 독립 | ❌ | ✅ 그래프 역추적 + Verifier 검증 |
| 학습 이력 누적 | ❌ | 🟡 점수만 | ✅ 개념별 약점 프로파일 |
| 자동 마이크로 러닝 | 🟡 수동 요청 | ❌ | ✅ 결손 노드에 자동 생성 |
| 이해 검증 게이트 | ❌ | 🟡 일괄 채점 | ✅ 퀴즈 통과 필수 |
| 멀티 에이전트 품질 보장 | ❌ | ❌ | ✅ Proposer-Verifier 루프 |

## 2. AI 활용 전략

### ■ 이 프로젝트에서 사용할 AI 도구(Claude Code, Cursor, Codex 등)와 모델은 무엇이며, 선택한 이유는 무엇인가요?

**개발 도구 (Human-AI 협업)**

- **Claude Code (Opus 4.6, 1M context)** — 메인 개발 파트너
  - **선택 이유**:
    - 1M 컨텍스트 → 대형 컴포넌트(700+ 줄 `knowledge-graph-canvas.tsx`)를 통째로 로드하면서도 안전한 수정 가능
    - Multi-step planning, 타입 체크 피드백 루프 (`tsc --noEmit`), 다중 파일 동시 edit 지원
    - 단순 코드 제너레이터가 아닌 **"추론하는 페어"** — 트레이드오프 논의, 리스크 사전 플래그, 대안 제시
    - 전체 개발 세션의 코드 작성 80% 이상 담당

- **v0.app** — 초기 UI 프로토타입 생성
  - **선택 이유**: 랜딩 페이지와 기본 컴포넌트의 초기 shell을 빠르게 생성. 이후 Claude Code로 마이그레이션하여 복잡한 로직/통합 작업 수행.

**런타임 AI 모델 (앱 내부)**

- **Claude Haiku 4.5** (`claude-haiku-4-5-20251001`) — 고빈도 / 저지연 작업
  - 오답 분석 Proposer, Verifier, Content Generator
  - 컨텍스트 검증 (그래프 생성 시 부족한 정보 추가 질문)
  - 챗봇 스트리밍 응답
  - **선택 이유**:
    - Sonnet 대비 **~5배 저렴** + 50ms~1s 응답 속도
    - 단순 분류/추출/요약/JSON 생성에서 Sonnet과 동등 수준 품질
    - 멀티 에이전트 루프처럼 호출 빈도가 높은 작업에 최적

- **Claude Sonnet 4.6** (`claude-sonnet-4-6`) — 저빈도 / 복잡 추론
  - 강의 텍스트 → 지식 그래프 JSON 생성
  - **선택 이유**:
    - 그래프 생성은 **구조적 추론**이 필요 (개념 간 의존 관계 파악 + 사이클 방지 + JSON 스키마 준수 + 노드 개수 자동 판단)
    - Haiku 테스트 시 prereq 누락이나 순환 참조가 발생 → Sonnet은 1회 호출로 안정적 결과
    - 호출 빈도가 낮아 비용 부담 적음

**비용 효율 설계**

| 모델 | 용도 | 호출 빈도 | 토큰/호출 |
|---|---|---|---|
| Haiku 4.5 | 분석 / 검증 / 콘텐츠 생성 / 챗봇 / 컨텍스트 검증 | **많음** (15) | 200-800 |
| Sonnet 4.6 | 그래프 생성 | **적음** (1) | 2000-4000 |

같은 작업을 전부 Sonnet으로 돌리는 경우 대비 **토큰 비용 약 70% 절감** (추정).

### A. Claude Code — 개발 파트너

**페르소나**: "페어 프로그래밍하는 시니어 엔지니어"

- 추측 대신 코드를 먼저 읽고 판단
- 트레이드오프를 명시적으로 논의
- 위험한 작업 전 확인 요청
- 단순 코드 생성이 아닌 설계 결정에 참여

**핵심 전략**

1. **3단계 루프** — 코드 먼저 쓰지 않음
   - 기능 제안 → 플래닝 대화 (트레이드오프 / 스코프 / 리스크) → 구현
2. **타입 체크 피드백 루프**
   - 변경마다 `npx tsc --noEmit` 자동 실행 → 에러 자가 수정
3. **다중 파일 동시 수정**
   - Edit tool로 한 번에 여러 파일 업데이트 (API route + 프론트 + 타입 export 동시에)
4. **지속적 요약**
   - 긴 세션에선 중간 요약으로 컨텍스트 관리 (CLAUDE.md 업데이트)
5. **CLAUDE.md 지침서**
   - 프로젝트 컨벤션 고정 (노드 타입, 엣지 ID 규칙, 색상 팔레트, 파일 경로) → 매 세션마다 재설명 방지

**제약 사항**

- `.env.local` 등 민감 파일 절대 수정 금지
- 사용자 확인 없이 `git push`, `git reset --hard`, `rm -rf` 금지
- 새 파일 생성보다 **기존 파일 편집 선호**
- 오버 엔지니어링 금지 (요구되지 않은 기능/추상화 추가 X)

### B. Claude Haiku 4.5 — 오답 분석 Multi-Agent Trio

3개 전문 에이전트로 분리, 각각 다른 페르소나와 프롬프트.

#### B-1. Proposer — "고속 그래프 역추적가"

- **역할**: 오답 텍스트에서 근본 원인 노드 1개 추론
- **프롬프트 전략**:
  - 노드 리스트를 **압축 파이프 포맷** (`4|행렬식|ad-bc 계산|prereqs:3`) → JSON 대비 ~40% 토큰 절감
  - Reasoning은 1-2문장으로 제한
  - 이전 Verifier 피드백이 있으면 `critique` 블록으로 주입하여 재분석 유도
  - `max_tokens: 200` (강력 제한)
- **제약**: JSON 외 텍스트 출력 금지, confidence는 0~1 실수

#### B-2. Verifier — "엄격한 교차 검증관"

- **역할**: Proposer 제안을 비판적으로 검토, 반박하거나 동의
- **프롬프트 전략**:
  - 제안된 노드의 prereq 목록을 **별도로 주입** → "이 선행 개념이 더 근본적 원인 아닌가?" 유도
  - **"애매하면 agree=true로 편향"** 지시 — false alarm 방지 (과도한 루프 방지)
  - `max_tokens: 200`
- **제약**: 반박 시에만 `critique` 작성 + 대안 노드 ID 제시

#### B-3. Content Generator — "최종 학습 콘텐츠 작가"

- **역할**: 확정된 결손 개념에 대해 explanation + 마이크로 러닝 + 퀴즈 생성
- **프롬프트 전략**:
  - **루프와 완전 분리** — Proposer/Verifier 루프 종료 후 **단 1회만 호출**
  - 이유: 루프마다 콘텐츠를 생성하면 버려지는 토큰이 많음. 최종 확정 후 한 번만 생성 → **버려지는 토큰 0**
  - `max_tokens: 800`

**에이전트 구성 방식** — Multi-Agent Harness Loop

```
[Step 1] 오류 패턴 분석

[Step 2] Proposer R1 (200tok)
   ↓
   confidence ≥ 0.8 ? → YES → Step 5
                      ↓ NO
[Step 3] Verifier R1 (200tok)
   ↓
   agree ? → YES → Step 5
          ↓ NO
[Step 4] Proposer R2 (200tok, with critique)
   ↓
   ... (최대 3 라운드)
   ↓
[Step 5] Content Generator (800tok, 1회)
   ↓
최종 결과
```

**조기 종료 4가지** (토큰 낭비 방지)
1. **confidence_high**: Proposer confidence ≥ 0.8 → 즉시 종료
2. **verifier_agreed**: Verifier 동의 → 종료
3. **converged**: 같은 노드 2회 연속 제안 → 수렴, 종료
4. **max_rounds**: 3 라운드 도달 → 종료

**실제 측정**: 조기 종료 덕분에 평균 1.5 라운드에서 종료 → 최악(3 라운드) 대비 50% 토큰 절감.

### C. Claude Sonnet 4.6 — 지식 그래프 생성

**페르소나**: "구조적 추론가 / 교육 과정 설계자"

- **역할**: 강의 텍스트 → 노드 + prereq 관계를 가진 DAG JSON 생성
- **프롬프트 전략**:
  - 출력 스키마를 프롬프트에 **예시 JSON으로 명시**
  - 사이클 금지 규칙 명시
  - 클라이언트 사이드 `detectCycle()` 함수로 검증 → 실패 시 에러 리턴 (Sonnet 재호출 없이)
- **컨텍스트 보강 패턴** (Haiku + Sonnet 협업):
  1. 사용자가 100자 미만 입력 → **Haiku**가 `/api/validate-context`로 부족한 정보 질문
  2. 최대 3라운드 Q&A로 컨텍스트 수집
  3. 충분해지면 → **Sonnet**이 1회 호출로 그래프 생성
  4. Sonnet 재호출 없이 한 번에 품질 확보

### D. 통합 데이터 흐름

```
User (Browser)
   ↓ HTTP POST
Next.js API Route (/api/analyze-error)
   ↓ Server-Sent Events (ReadableStream)
Agent Harness Loop (Server)
   ↓ SSE events, 실시간 스트리밍

   ↓
Client React State (TextDecoder stream reader)
   ↓
KnowledgeGraphCanvas    (실시간 스텝 overlay)

```

**핵심 설계 포인트**

- SSE로 **분석 중간 과정이 실시간으로 UI에 반영** → 사용자는 "AI가 지금 뭘 하고 있는지" 인지 가능 (블랙박스 방지)
- `AgentTraceEntry` 유니온 타입을 API route에서 export → 프론트/백 타입 안전성 확보
- 트레이스 전체는 `ErrorLog.result.agentTrace`에 저장 → 나중에 복원 가능

### ■ 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략

### 토큰 낭비 최소화

**① 모델 계층화 (가장 큰 효과)**

- 복잡 추론(그래프 생성) → **Sonnet 4.6**
- 분류/추출/요약/챗 → **Haiku 4.5**
- 모든 작업을 Sonnet으로 돌리는 경우 대비 **토큰 비용 ~70% 절감** (추정)

**② Two-phase Proposer (핵심 최적화)**

- **Before**: Proposer가 매 루프마다 explanation + microLearning + quiz까지 생성 (~800tok/호출)
- **After**: 루프 중엔 **진단만 (100tok)**, 최종 확정 후 Content Generator 1회 호출 (800tok)
- **효과**: 3라운드 루프 시
  - 기존: 3 × 800tok = **2,400tok**
  - 현재: 3 × 100 + 800 = **1,100tok** (약 54% 절감)

**③ 압축 프롬프트 포맷**

- 노드 리스트: `id|label|desc|prereqs` 파이프 구분자
- 기존 JSON (`{"id":"4","label":"행렬식",...}`) 대비 **~40% 토큰 절감**
- 프롬프트 전체에서 반복 사용되는 구조 → 효과 극대화

**④ 조기 종료 4가지**

confidence 임계값 / verifier 동의 / 수렴 감지 / max rounds → **평균 1.5 라운드**에서 종료.

**⑤ max_tokens 엄격 제한**

| Agent | max_tokens |
|---|---|
| Proposer | 200 |
| Verifier | 200 |
| Content Generator | 800 |

무한 생성 가능성 원천 차단.

**⑥ 컨텍스트 검증으로 Sonnet 낭비 방지**

그래프 생성 시 입력이 부족하면 **Haiku가 질문 (200tok × 최대 3회)** → 충분해진 후 Sonnet 호출. Sonnet 재호출 없이 품질 확보.

### 재현성

**① 프롬프트 Git 버전 관리**

모든 프롬프트 문자열이 소스 코드에 있음 → `git diff`로 프롬프트 변경사항 추적. 회귀 테스트 시 커밋 SHA로 특정 버전 복원 가능.

**② 전체 에이전트 trace 영속화**

- Quiz 옵션 셔플은 Fisher-Yates로 서버에서 실행 (편향 없는 셔플)
- 클라이언트에서도 한 번 더 셔플 (하드코딩 fallback 대응)
- `useMemo(..., [selectedNode.id])`로 렌더마다 재셔플 방지

**④ 환경 분리**

- `.env.local` gitignore 엄격 관리
- Vercel 환경 변수로 배포 (Production / Preview / Development 분리)
- API 키 누출 방지 검증 (`git log -p --all -- '*.env*'`)
